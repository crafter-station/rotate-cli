# Neon connection strings — ownership detection

Target env vars: `DATABASE_URL`, `POSTGRES_URL`, `POSTGRES_PRISMA_URL`, `POSTGRES_URL_NON_POOLING`, `POSTGRES_URL_NO_SSL`, `DATABASE_URL_UNPOOLED`, `NEON_POSTGRES_URL`, and any custom `*_POSTGRES_URL` / `*_DATABASE_URL` pattern.

This file covers **postgres connection strings only**. For `NEON_API_KEY` (opaque tokens), see `./neon.md` — that's a different secret class with different handling.

## Summary

A Neon connection string is `postgresql://{role}:{password}@ep-{adjective}-{noun}-{random}[-pooler].{region}.{cloud}.neon.tech/{db}?sslmode=require`. The `ep-xxx-yyy-zzzzzz` component is the **endpoint id**, which is globally unique across the Neon platform — it's the routing token used by Neon's proxy to resolve the connection to a specific compute instance.

Because endpoint ids are globally unique, `ownedBy()` is pure string parsing + one lookup in a cached reverse index. Zero network calls at query time; certainty is high; no candidate probing required.

## Strategy

### Format-decode + cached reverse index

1. Regex the hostname out of the connection string.
2. Strip the optional `-pooler` suffix.
3. Extract `endpoint_id = "ep-..."`.
4. Look up in `ctx.endpointToProject` (built lazily at rotate-cli setup).
5. Hit → check `project.org_id` against `ctx.knownOrgIds` → `self` or `other`. Miss → `unknown` (likely foreign, but could be a project the admin key cannot see).

The reverse index is built once per rotate-cli session by iterating admin's projects and listing endpoints per project. Neon has no `GET /endpoints/{ep-xxx}` reverse lookup, so the index must be built by forward iteration.

### Globally unique vs namespaced: empirical confirmation

Neon's docs do not explicitly state the uniqueness guarantee for endpoint ids, but three lines of evidence confirm global uniqueness:

1. **Heroku-like naming scheme**: `ep-{adjective}-{noun}-{6-char-hex}` with ~1000 adjectives × 1000 nouns × 16^6 random tail ≈ 10^10 combinations, generated by Neon's central ID service. Collision probability is negligible.
2. **Hostname resolution**: the proxy at `*.neon.tech` uses the endpoint_id alone (no org prefix) to route traffic. If ids were org-namespaced, DNS/SNI resolution would require an org qualifier — which is not in the hostname.
3. **API surface**: `GET /projects/{project_id}/endpoints/{endpoint_id}` requires `project_id`, but that's a documentation/permission artifact — an endpoint id alone unambiguously identifies a compute. The project_id in the path is authorization scope, not disambiguation.

Therefore `ep-xxx` → exactly one project → exactly one org. Safe to treat as primary key.

### Why introspection is infeasible

The connection string is a postgres credential, not an API bearer. Using it requires a live TCP+TLS handshake and a `SELECT` against `pg_catalog`. That's expensive, slow (~500ms+), and reveals nothing about org membership — the database role has no knowledge of the Neon control plane. `SELECT current_database()` returns the db name, which is project-adjacent but not authoritative.

Format-decode is strictly superior: cheaper, faster, higher-signal.

## Endpoints used (bootstrap only)

Base: `https://console.neon.tech/api/v2` (also `https://api.neon.tech/v2`).

```bash
# 1. List orgs the admin can see
curl https://console.neon.tech/api/v2/users/me/organizations \
  -H "Authorization: Bearer $NEON_ADMIN_KEY"

# 2. List projects (optionally filtered by org)
curl "https://console.neon.tech/api/v2/projects?org_id=org-xxx&limit=100&cursor=..." \
  -H "Authorization: Bearer $NEON_ADMIN_KEY"

# Response includes projects[] with { id, name, org_id, region_id, ... }.
# If admin uses a personal API key, org_id must be passed to scope to each org.
# If admin uses an org-scoped key, the key is already bound; omit org_id.

# 3. Per project, list endpoints
curl "https://console.neon.tech/api/v2/projects/$PROJECT_ID/endpoints" \
  -H "Authorization: Bearer $NEON_ADMIN_KEY"

# Response: endpoints[] with { id: "ep-xxx-yyy-zzzzzz", host, project_id, branch_id, type, region_id, ... }
# `host` is the routable FQDN; `id` is what's in the connection string hostname.
```

Build `Map<endpoint_id, { project_id, org_id, host }>` once, reuse for all candidates.

## Implementation hints (pseudocode)

```ts
// adapter-neon/ownership-connection.ts

type Ctx = {
  knownOrgIds: Set<string>;
  endpointToProject: Map<string, { projectId: string; orgId: string }>;
};

const HOSTNAME_RE =
  /@((?:ep-[a-z]+-[a-z]+-[a-z0-9]+))(?:-pooler)?\.(?:[a-z0-9-]+\.)?(?:aws|gcp|azure)\.neon\.tech/i;

export function ownedBy(
  secret: string,
  ctx: Ctx,
): "self" | "other" | "unknown" {
  if (!secret.startsWith("postgres://") && !secret.startsWith("postgresql://")) {
    return "unknown";
  }

  const m = secret.match(HOSTNAME_RE);
  if (!m) return "unknown";

  const endpointId = m[1].toLowerCase();
  const hit = ctx.endpointToProject.get(endpointId);
  if (!hit) return "unknown"; // admin can't see this project — likely "other"

  return ctx.knownOrgIds.has(hit.orgId) ? "self" : "other";
}

export async function seedCtx(adminKey: string): Promise<Ctx> {
  const headers = { Authorization: `Bearer ${adminKey}` };
  const base = "https://console.neon.tech/api/v2";

  // Step 1: orgs
  const orgs = await fetchJson(`${base}/users/me/organizations`, headers);
  const knownOrgIds = new Set<string>(orgs.organizations.map((o: any) => o.id));
  // also include "personal" scope — personal projects have org_id === "" or null
  knownOrgIds.add(""); knownOrgIds.add("personal");

  // Step 2: projects per org + per-personal
  const projects: Array<{ id: string; org_id: string }> = [];
  for (const orgId of knownOrgIds) {
    let cursor: string | undefined;
    do {
      const qs = new URLSearchParams({ limit: "100" });
      if (orgId && orgId !== "personal" && orgId !== "") qs.set("org_id", orgId);
      if (cursor) qs.set("cursor", cursor);
      const body = await fetchJson(`${base}/projects?${qs}`, headers);
      for (const p of body.projects ?? []) {
        projects.push({ id: p.id, org_id: p.org_id ?? "personal" });
      }
      cursor = body.pagination?.cursor;
    } while (cursor);
  }

  // Step 3: endpoints per project
  const endpointToProject = new Map<string, { projectId: string; orgId: string }>();
  for (const p of projects) {
    const body = await fetchJson(`${base}/projects/${p.id}/endpoints`, headers);
    for (const ep of body.endpoints ?? []) {
      endpointToProject.set(ep.id, { projectId: p.id, orgId: p.org_id });
    }
  }

  return { knownOrgIds, endpointToProject };
}
```

## Edge cases

- **Pooler suffix**: `ep-cool-darkness-a1b2c3d4-pooler.us-east-2.aws.neon.tech`. The `-pooler` is a routing hint (pgbouncer in front of the compute), not part of the id. Regex strips it with the optional `(?:-pooler)?` group.
- **Branched databases**: each branch has its own endpoint id. A rotation audit might see 3 connection strings pointing to the same project but different branches (main, preview, dev) — all three endpoints live in the reverse index, all resolve to same `project_id` → same `org_id`. Correct.
- **Pending endpoints / suspended computes**: a suspended compute keeps its `endpoint_id` in the control plane and in the reverse index. Connection string still decodes correctly; ownership detection still returns `self`.
- **Deleted endpoint**: the id is removed from `GET /projects/{id}/endpoints`. A connection string pointing to a deleted endpoint → miss → `unknown`. That's the correct answer; rotate-cli should flag it for human review (likely stale env var).
- **Custom hostnames / vanity URLs**: Neon does not support customer CNAMEs pointing to `neon.tech`. The hostname format is fixed. No need to handle vanity mappings.
- **Self-hosted Neon / Neon Proxy**: the hostname pattern differs (e.g. `db.internal.company.com`). Regex fails → `unknown`. Adapter should fall through to the admin's human-review queue. This is a tiny minority case.
- **Pre-pooled vs pooled connection strings**: Vercel often sets both `POSTGRES_URL` (pooled) and `POSTGRES_URL_NON_POOLING` (direct). Both decode to the same endpoint id after stripping `-pooler`.
- **`options` or `host=` query params**: rare, but some legacy connection strings embed `options=project%3Dep-...` for SNI workarounds. The `ep-xxx` in the query param is the same id — apply the regex to the full string, not just the hostname fragment.
- **Region in hostname**: `.us-east-2.aws.neon.tech`, `.eu-central-1.aws.neon.tech`, `.c-2.us-east-2.aws.neon.tech`. Region is metadata, not identity. The regex accepts any region slug before `.aws|gcp|azure.neon.tech`.
- **Password leakage**: the connection string includes the role password in cleartext. Ownership detection should never log the full string — log only the `endpoint_id` + masked host. Treat the candidate as tier-0 sensitive.
- **`neondb_owner` vs custom roles**: the role name in the URL is not tied to ownership. Rotation targets the *password* (via `PATCH /projects/{id}/branches/{branch_id}/roles/{role_name}/reset_password`), not the endpoint. So ownership detection is upstream of rotation and independent of the role.
- **Personal account vs org projects**: in `GET /projects`, personal projects have `org_id` as empty string or null. Normalize to a sentinel ("personal") in the reverse index. Hunter's context marks "personal" as owned.
- **Cross-org projects**: Neon supports transferring a project between orgs. Between transfer and next index refresh, a connection string might resolve to a stale org mapping. Refresh cadence: rebuild on each `rotate status` run, or TTL the index at 1h.
- **Hostname with trailing dot**: `ep-xxx-yyy.us-east-2.aws.neon.tech.` (valid FQDN, rare). The regex tolerates it because the trailing dot doesn't match the capture group.
- **`.env` files with quotes**: `DATABASE_URL="postgres://..."`. Trim quotes before regex.

## Verdict summary

| Secret shape | Method | Cost | Certainty |
|---|---|---|---|
| `postgresql://...@ep-xxx-...neon.tech/...` | Regex endpoint id, lookup in reverse index | 0 calls (warm) | High |
| Pooled variant (`-pooler`) | Same, strip `-pooler` | 0 calls (warm) | High |
| Legacy SNI `options=project%3Dep-xxx` | Same regex on query param | 0 calls (warm) | High |
| Non-Neon postgres URL | Regex fails | — | N/A (delegate to another adapter) |

**Why high certainty**: endpoint ids are globally unique, immutable for the lifetime of the compute, and carried verbatim in the DNS hostname. Unlike opaque API keys, they leak their project affiliation by construction. The only ambiguity is "does admin have visibility into this project?" — a `miss` in the reverse index means either the endpoint is foreign (owned by another org) or the admin's visibility is incomplete. Surface both outcomes as `unknown` on first pass; rotate-cli's human reviewer resolves the disambiguation.
